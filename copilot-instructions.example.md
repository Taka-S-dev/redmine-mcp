# Copilot CLI 指示書テンプレート（redmine-mcp）

> **使い方**
> 1. このファイルを `copilot-instructions.md` にコピー（リネーム）
>    PowerShell: `Copy-Item copilot-instructions.example.md copilot-instructions.md`
> 2. **見出しのタグを見て、編集する章だけ手を入れる**：
>    - `【そのまま】` … 触らない。汎用ノウハウなので編集不要
>    - `【記入推奨】` … 自社に合わせて埋めると効果大
>    - `【任意】` … 必要な運用があれば書く（無ければ章ごと消してよい）
> 3. 編集は `<...>` のプレースホルダを実際の値に置き換えるだけ
> 4. 埋めた `copilot-instructions.md` は `.gitignore` で追跡対象外（会社情報を含むため）
> 5. Copilot CLI をプロジェクトディレクトリで起動すると自動で読み込まれる
>
> Claude Code で使う場合は、同じ内容を `CLAUDE.md` に置けばよい。

## このプロジェクトについて　【そのまま】

`redmine` MCP サーバー経由で Redmine のチケットを検索・集計・分析する。
**読み取り専用** —— チケットの作成・更新・削除はできない（設計上の意図）。

## 用語マッピング　【記入推奨】

ユーザーが下表の左側の言葉を使ったら、右側のフィールドで検索すること。
会社の Redmine は GUI ラベルがカスタマイズされていることがあり、
**画面表示名と内部フィールド名が一致しない**点に注意。

▼ 下表の `<...>` の行を、自社の実フィールド名に置き換える（行は増やしてよい）

| ユーザーの言葉 | 実フィールド | MCP での指定方法 |
|---|---|---|
| 内容 / 本文 / 詳細 / 概要 | 標準 description | `search_issues` の `description_contains` |
| 件名 / 題名 / タイトル | 標準 subject | `search_issues` の `subject_contains` |
| コメント / 注記 / やりとり | journals | `search_issues` の `notes_contains` |
| 担当 / アサイン | assigned_to | `search_issues` の `assigned_to` |
| `<自社用語1>` | `<カスタムフィールド名>` | `custom_fields: { "<CF名>": "<値>" }` |
| `<自社用語2>` | `<カスタムフィールド名>` | `custom_fields: { "<CF名>": "<値>" }` |

（上 4 行は標準フィールドなのでそのまま。`<...>` の行だけ編集）
実際のカスタムフィールド名は `describe_schema` または `list_custom_fields` で確認できる。

## チケットの親子構造　【任意】

> 親子運用をしていなければ、この章は丸ごと削除してよい。

Redmine のチケットは親子構造を持てる。自社で親チケット・子チケットを
それぞれ特定の名前で呼んでいる場合、その対応をここで定義する。

▼ 下のブロックの `<...>` を自社の呼び方に置き換える

```
- 親チケット = <親チケットの自社での呼び方>
- 子チケット = <子チケットの自社での呼び方>

指示の解釈ルール:
- 「<親の呼び方> #N の <子の呼び方>」
  → search_issues({ parent_id: N }) で子チケットを取得
- 「この <子の呼び方> はどの <親の呼び方> か」
  → get_issue で応答の parent フィールドを見る
- 一覧で親と子を区別する
  → search_issues の結果の parent_id を見る（null なら親、値があれば子）
```

`search_issues` / `export_issues_csv` の出力には `parent_id` が含まれるので、
これを使えば親・子の判別や、親配下の子チケット集計ができる。

## 検索のお作法　【そのまま】

- **「完了」はステータス名で指定**: `status: "完了"`（または自社の完了ステータス名）。
  `status: "closed"` は Redmine の is_closed フラグ依存で、設定次第では動かない。
- **「期限切れ」は `overdue: true`** を使う（status を open に強制し、期日 < 今日 で絞る）。
- **件数だけ知りたいとき**は `count_only: true`（トークン節約）。
- **集計**（担当者別・カテゴリ別など）は `search_issues` で全件取らず **`aggregate_issues`** を使う。
- **工数の集計**は `list_time_entries`。親チケット配下をまとめるなら `parent_issue_id`。
- カスタムフィールドの値で**部分一致**したいときは `custom_field_match: "partial"`。
- リンクを提示するときは応答の `url` フィールドをそのまま使う（ベース URL を尋ねない）。
- **CSV / Excel 出力**は `export_issues_csv` を使う。LLM に表を生成させない（遅い・文字化けする）。

## 検索の振り分け方針　【そのまま】

ユーザーの指示が曖昧なときは、ツールの選び方を間違えると 0 件になる。以下の方針で振り分ける：

- **キーワードが曖昧／どのフィールドか特定できない** → `quick_search`（全文検索）を使う。
- **「〜が含まれる」「〜という言葉のあるチケット」** → `quick_search`（どのフィールドかを限定しない検索）。
- **「〜について」「〜まわり」「〜関連」「〜っぽい」のような曖昧表現** → `quick_search`。
- **ステータス・担当者・期間など条件が明確** → `search_issues`。
- **`search_issues` で 0 件だったら、あきらめずに `quick_search` で再検索**する。
- **カスタムフィールドの値が一部しか分からない** → `custom_field_match: "partial"`。
- 完全一致で見つからないときは、部分一致や全文検索に切り替えてから「無い」と結論づける。

## トラッカー別・回答フィールド　【任意】

> 回答に出るフィールドを統一したくなければ、この章は丸ごと削除してよい。

チケットについて回答するとき、**トラッカーごとに「必ず示すフィールドと順序」を
決めておくと、回答内容が毎回ブレない**。

▼ 下のブロックの `<...>` を自社のトラッカー名・フィールドに置き換える（トラッカーは増やしてよい）

```
### <トラッカー名A>
ID / 件名 / ステータス / 担当者 / 期日 / <CF名1> / <CF名2>

### <トラッカー名B>
ID / 件名 / ステータス / 進捗率 / 担当者
```

適用ルール（この部分はそのまま使ってよい）:

- チケットを 1 件説明するときは、そのトラッカーの定義順でフィールドを示す。
- 定義に無いトラッカーは、`ID / 件名 / ステータス / 担当者 / 期日` を基本とする。
- ユーザーが特定のフィールドを名指しで聞いた場合は、定義に関わらずそれを答える。
- CF が未設定（空）の場合も、フィールド名は出して値を「（未設定）」と示す
  ＝「何が空欄か」も分かるようにする。

## CSV エクスポートの標準列　【任意】

> CSV の列構成を固定したくなければ、この章は丸ごと削除してよい。

`export_issues_csv` を使うとき、`fields` を省略したときに使う「標準の列構成」を
ここで定義しておくと、毎回同じレイアウトの CSV が出る。

▼ 下の【記入例】を参考に、`標準列:` の行を自社で使いたい列構成に書き換える。
列名は標準フィールド（`id` `url` `subject` `status` `priority` `assigned_to`
`done_ratio` `start_date` `due_date` 等）と、カスタムフィールド `cf:<CF名>` が使える。
書いた順に CSV の列が並ぶ。

【記入例】（このまま使ってもよい。CF を足したいときは下の「トラッカー別」を参考に）

```
標準列: id / url / subject / status / priority / assigned_to / due_date

# トラッカーごとに列を変えたい場合だけ、以下のように追記する（任意）:
# <トラッカー名A> の CSV 列:
#   id / url / subject / cf:<CF名1> / cf:<CF名2> / status / assigned_to / due_date
```

適用ルール（この部分はそのまま使ってよい）:

- 「CSV にして」「Excel で出して」と言われたら、上記の標準列を `fields` に指定して
  `export_issues_csv` を呼ぶ。
- トラッカー別の定義があれば、そのトラッカーの列構成を優先する。
- ユーザーが列を明示的に指定したら、それを最優先する。
- `url` 列は基本入れておく（Excel からチケットを直接開けるため）。

## やってはいけないこと　【そのまま】

- Redmine への書き込み操作（MCP は読み取り専用、そもそも不可）
- `REDMINE_PROJECTS` でスコープを絞っている場合、スコープ外プロジェクトの詮索
- ベース URL や API キーをユーザーに尋ねること（`.env` に設定済み）
