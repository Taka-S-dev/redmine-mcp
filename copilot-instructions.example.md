> **使い方**
>
> 1. このファイルを `copilot-instructions.md` にコピー（リネーム）
>    PowerShell: `Copy-Item copilot-instructions.example.md copilot-instructions.md`
> 2. **見出しのタグを見て、編集する章だけ手を入れる**：
>    - `【そのまま】` … 触らない。汎用ノウハウなので編集不要
>    - `【記入推奨】` … この環境に合わせて埋めると効果大

> 5. Copilot CLI をプロジェクトディレクトリで起動すると自動で読み込まれる
>
> Claude Code で使う場合は、同じ内容を `CLAUDE.md` に置けばよい。

## このプロジェクトについて　【そのまま】

`redmine` MCP サーバー経由で Redmine のチケットを検索・集計・分析する。
**読み取り専用** —— チケットの作成・更新・削除はできない（設計上の意図）。

## 用語マッピング　【記入推奨】

ユーザーが下表の左側の言葉を使ったら、右側のフィールドで検索すること。
Redmine は GUI ラベルがカスタマイズされていることがあり、
**画面表示名と内部フィールド名が一致しない**点に注意。

▼ 下表の `<...>` の行を、この環境の実フィールド名に置き換える（行は増やしてよい）

| ユーザーの言葉             | 実フィールド             | MCP での指定方法                          |
| -------------------------- | ------------------------ | ----------------------------------------- |
| 内容 / 本文 / 詳細 / 概要  | 標準 description         | `search_issues` の `description_contains` |
| 件名 / 題名 / タイトル     | 標準 subject             | `search_issues` の `subject_contains`     |
| コメント / 注記 / やりとり | journals                 | `search_issues` の `notes_contains`       |
| 担当 / アサイン            | assigned_to              | `search_issues` の `assigned_to`          |
| `<用語1>`                  | `<カスタムフィールド名>` | `custom_fields: { "<CF名>": "<値>" }`     |
| `<用語2>`                  | `<カスタムフィールド名>` | `custom_fields: { "<CF名>": "<値>" }`     |

（上 4 行は標準フィールドなのでそのまま。`<...>` の行だけ編集）
実際のカスタムフィールド名は `describe_schema` または `list_custom_fields` で確認できる。

## チケットの親子構造　【任意】

> 親子運用をしていなければ、この章は丸ごと削除してよい。

Redmine のチケットは親子構造を持てる。プロジェクトで親チケット・子チケットを
それぞれ特定の名前で呼んでいる場合、その対応をここで定義する。

▼ 下のブロックの `<...>` を実際の呼び方に置き換える

```
- 親チケット = <親チケットの呼び方>
- 子チケット = <子チケットの呼び方>

指示の解釈ルール:
- 「<親の呼び方> #N の <子の呼び方>」
  → search_issues({ parent_id: N }) で子チケットを取得
- 「この <子の呼び方> はどの <親の呼び方> か」
  → get_issue で応答の parent フィールドを見る
- 一覧で親と子を区別する
  → search_issues の結果の parent_id を見る（null なら親、値があれば子）
```

`search_issues` / `export_issues_csv` の出力には `parent_id` が含まれるので、
これを使えば親・子の判別や、親配下の子チケット集計ができる。

## 検索のお作法　【そのまま】

- **「完了」はステータス名で指定**: `status: "完了"`（またはこの Redmine の完了ステータス名）。
  `status: "closed"` は Redmine の is_closed フラグ依存で、設定次第では動かない。
- **「期限切れ」は `overdue: true`** を使う（status を open に強制し、期日 < 今日 で絞る）。
- **件数だけ知りたいとき**は `count_only: true`（トークン節約）。
- **集計**（担当者別・カテゴリ別など）は `search_issues` で全件取らず **`aggregate_issues`** を使う。
- **工数の集計**は `list_time_entries`。親チケット配下をまとめるなら `parent_issue_id`。
- カスタムフィールドの値で**部分一致**したいときは `custom_field_match: "partial"`。
- リンクを提示するときは応答の `url` フィールドをそのまま使う（ベース URL を尋ねない）。
- **CSV / Excel 出力**は `export_issues_csv` を使う。LLM に表を生成させない（遅い・文字化けする）。

## 検索の振り分け方針　【そのまま】

ユーザーの指示が曖昧なときは、ツールの選び方を間違えると 0 件になる。以下の方針で振り分ける：

- **キーワードが曖昧／どのフィールドか特定できない** → `quick_search`（全文検索）を使う。
- **「〜が含まれる」「〜という言葉のあるチケット」** → `quick_search`（どのフィールドかを限定しない検索）。
- **「〜について」「〜まわり」「〜関連」「〜っぽい」のような曖昧表現** → `quick_search`。
- **ステータス・担当者・期間など条件が明確** → `search_issues`。
- **`search_issues` で 0 件だったら、あきらめずに `quick_search` で再検索**する。
- **カスタムフィールドの値が一部しか分からない** → `custom_field_match: "partial"`。
- 完全一致で見つからないときは、部分一致や全文検索に切り替えてから「無い」と結論づける。

## トラッカー別・回答フィールド　【任意】

> 回答に出るフィールドを統一したくなければ、この章は丸ごと削除してよい。

チケットについて回答するとき、**トラッカーごとに「必ず示すフィールドと順序」を
決めておくと、回答内容が毎回ブレない**。

▼ 下のブロックの `<...>` を実際のトラッカー名・フィールドに置き換える（トラッカーは増やしてよい）

```
### <トラッカー名A>
ID / 件名 / ステータス / 担当者 / 期日 / <CF名1> / <CF名2>

### <トラッカー名B>
ID / 件名 / ステータス / 進捗率 / 担当者
```

適用ルール（この部分はそのまま使ってよい）:

- チケットを 1 件説明するときは、そのトラッカーの定義順でフィールドを示す。
- 定義に無いトラッカーは、`ID / 件名 / ステータス / 担当者 / 期日` を基本とする。
- ユーザーが特定のフィールドを名指しで聞いた場合は、定義に関わらずそれを答える。
- CF が未設定（空）の場合も、フィールド名は出して値を「（未設定）」と示す
  ＝「何が空欄か」も分かるようにする。

## CSV エクスポートの標準列　【任意】

> CSV の列構成を固定したくなければ、この章は丸ごと削除してよい。

`export_issues_csv` を使うとき、`fields` を省略したときに使う「標準の列構成」を
ここで定義しておくと、毎回同じレイアウトの CSV が出る。

▼ 下の【記入例】を参考に、`標準列:` の行を使いたい列構成に書き換える。
列名は標準フィールド（`id` `url` `subject` `status` `priority` `assigned_to`
`done_ratio` `start_date` `due_date` 等）と、カスタムフィールド `cf:<CF名>` が使える。
書いた順に CSV の列が並ぶ。

【記入例】（このまま使ってもよい。CF を足したいときは下の「トラッカー別」を参考に）

```
標準列: id / url / subject / status / priority / assigned_to / due_date

# トラッカーごとに列を変えたい場合だけ、以下のように追記する（任意）:
# <トラッカー名A> の CSV 列:
#   id / url / subject / cf:<CF名1> / cf:<CF名2> / status / assigned_to / due_date
```

適用ルール（この部分はそのまま使ってよい）:

- 「CSV にして」「Excel で出して」と言われたら、上記の標準列を `fields` に指定して
  `export_issues_csv` を呼ぶ。
- トラッカー別の定義があれば、そのトラッカーの列構成を優先する。
- ユーザーが列を明示的に指定したら、それを最優先する。
- `url` 列は基本入れておく（Excel からチケットを直接開けるため）。

## チケットレビュー（review_issue の使い方）　【任意】

> 「チケットの記入もれ・内容の妥当性をチェックする」運用が無ければ、この章は丸ごと削除してよい。

`review_issue` ツールは、チケット 1 件の **記入もれ**（決定論）と **内部矛盾**（決定論）を
機械判定し、**内容の妥当性レビュー**（定性判断）の材料を返す。本章で「どのトラッカーで何を
必須にするか」「どの観点で内容を判定するか」を定義しておくと、AI が毎回同じ品質でレビューする。

### 発動条件（この部分はそのまま使ってよい）

ユーザーが「チケット #X をレビューして」「#X の記入内容チェックして」のように言ったら、
`review_issue` を必ず呼ぶ。

### 同時に取得するもの（include\_\* の使い分け）

▼ 下を参考に実際の運用に合わせる（消してもよい）

- **親チケットをレビューするとき**は `include_children: true` で子チケットも含める
- **子チケットをレビューするとき**は、応答の `parent_id` を見て、必要なら `get_issue` で親も取得して比較する
- コメント履歴の整合確認が要るトラッカーは `include_journals: true`
- 添付ファイルが審査対象になるなら `include_attachments: true`

### レビューのルール（必須項目 + 内容観点）

▼ フィールド単位で「いつ・必須か・どう判定するか」を 1 行で書く。観点を増やすときは行追加するだけ。

**列の意味**:

- **適用** … この行が効く場面。`全部` / `<トラッカー名>`（複数なら `A / B` とスラッシュ区切り）/ `子のレビュー時` / `親のレビュー時` / `親子セット`
- **必須** … `✓` なら `required_fields` パラメータに入って機械的に空欄チェック
- **観点** … LLM が定性判定する内容。`-` なら定性判定スキップ

| フィールド            | 適用                                | 必須 | 観点                                                                |
| --------------------- | ----------------------------------- | ---- | ------------------------------------------------------------------- |
| 件名                  | 全部                                | ✓    | `<例: 対象と現象が具体的に書かれているか>`                          |
| 説明                  | `<トラッカー名A>`                   | ✓    | `<例: 【現象】【再現手順】【期待結果】の3要素が揃っているか>`       |
| 担当者                | `<トラッカー名A> / <トラッカー名B>` | ✓    | -                                                                   |
| 期日                  | `<トラッカー名A>`                   | ✓    | -                                                                   |
| `cf:<CF名1>`          | `<トラッカー名A>`                   | ✓    | `<例: 推測でなく特定済みの記述か>`                                  |
| `cf:<CF名2>`          | `<トラッカー名A>`                   | ✓    | `<例: cf:<CF名1> と論理的に対応しているか>`                         |
| `cf:<CF名3>`          | `<トラッカー名B>`                   | ✓    | -                                                                   |
| ステータス × コメント | 全部                                | -    | `<例: ステータス変更とコメント内容の時系列が矛盾しないか>`          |
| 親チケットとの整合    | 子のレビュー時                      | -    | `<例: 子の対応内容が親で示された要件・課題への解決か>`              |
| 子チケット群の整合    | 親のレビュー時                      | -    | `<例: 親の各観点に対応する子チケットが揃っているか / 漏れが無いか>` |
| 添付ファイル          | `<該当トラッカー>`                  | -    | `<例: チェックリストの日付が期日と整合しているか>`                  |

**フィールド名の表記**:

- 標準フィールドは英字キー: `subject` / `description` / `assigned_to` / `due_date` / `start_date` / `estimated_hours` / `category` / `fixed_version` / `parent`
- CF は `cf:<CF名>` 形式（GUI で見えてるフィールド名そのままでOK）
- 「親チケットとの整合」「子チケット群の整合」「添付ファイル」「ステータス × コメント」は "観点専用" のメタ行（必須=- で必須項目化はしない）

▼ AI の解釈ルール（編集不要）

**大原則**: **表に書かれていないフィールドは「チェックしない / 任意」扱い**。必須化も定性判定もしない。書いたものだけ気にする。

1. レビュー対象トラッカー T について、`適用 = 全部` または `適用` に T が含まれる行を採用
2. 採用行のうち `必須 = ✓` のフィールドを集めて `required_fields` パラメータに渡す
3. 採用行のうち `観点 ≠ -` の観点を LLM が定性判定に使う
4. `適用 = 子のレビュー時` の行は対象に親があるときに追加適用（応答の `parent_id` 経由で親を取得）
5. `適用 = 親のレビュー時` の行は対象を親としてレビューするときに適用（`include_children: true` 経由）
6. 「添付ファイル」行が採用されたら `include_attachments: true` で取得 → 必要に応じ `download_attachment` で中身を確認

### 出力フォーマット（この部分はそのまま使ってよい）

**結論先 → 本題に絞る → 補足は後置** の 3 段で構成する。書きたいことが多くても結論を埋もれさせない・本題と副次観察を混ぜない。

1. **冒頭で「総合判定: OK / 要修正」を 1 行**（最初に結論を示す）
2. **チェック項目だけに絞った結果**（聞かれたルール = レビューのルール表で採用された行への直接の答えだけ。表で簡潔に。重大度マーク ❌（要対応）/ ⚠️（要確認）/ ✅（問題なし）を付ける）
3. **「他に気になった点」**（ファイル設定の typo、未置換プレースホルダ、副次的な発見等は最後に区切って後置。本題と混ぜない。無ければ書かない）

### AI の想定挙動（参考・編集不要）

**[単一チケットのレビュー]**

```
ユーザ: 「#123 をレビューして」
AI:    get_issue でトラッカーを確認 → <トラッカー名A> と判明
AI:    review_issue(
         issue_id: 123,
         required_fields: ["subject","description","assigned_to","due_date",
                           "cf:<CF名1>","cf:<CF名2>"],
         include_journals: true
       ) を呼ぶ
AI:    completeness / consistency_flags を「記入もれ」「内部矛盾」に整理し、
       issue 本体 + journals を読んで「内容の妥当性」観点を上から評価
AI:    3 セクション形式でユーザにレポート
```

**[子チケットのレビュー（親との整合まで見る）]**

```
ユーザ: 「子チケット #234 をレビューして」
AI:    review_issue(issue_id: 234, required_fields: [...], include_journals: true)
AI:    応答の parent_id (= 100 とする) を確認 → get_issue(100) で親を取得
AI:    子の完全性 / 矛盾を整理 + 「親チケットとの整合」行の観点で親と比較
AI:    3 セクションでレポート（「内容の妥当性」に親整合の結果を含める）
```

**[親チケット + 子全件のセットレビュー]**

```
ユーザ: 「親 #500 とその子全部をまとめてレビューして」
AI:    review_issue(issue_id: 500, include_children: true, include_journals: true)
AI:    親自身を評価 + children の id 一覧を取得
AI:    各 child id について review_issue を呼んで個別評価
       （child のトラッカーに応じた required_fields で）
AI:    最後に「子チケット群の整合」行の観点で集合レビュー
AI:    親 → 各子 → 親子整合 の順でレポート
```

## やってはいけないこと　【そのまま】

- Redmine への書き込み操作（MCP は読み取り専用、そもそも不可）
- `REDMINE_PROJECTS` でスコープを絞っている場合、スコープ外プロジェクトの詮索
- ベース URL や API キーをユーザーに尋ねること（`.env` に設定済み）
